================================================================================
内核调度 OS 线程 / 线程池线程 / Python 线程  ——  三层调度关系总结
================================================================================

【核心一句话】
内核眼里从来只有两样东西：OS 线程，以及它此刻的状态（可运行 or 睡眠）。
线程池、任务队列、GIL 全是用户态在上面叠的协调逻辑，内核一概不知；它们能
影响调度的唯一方式，就是通过“睡眠 ↔ 唤醒”去改变线程的可运行状态。


================================================================================
一、三个层级各管各的，不在同一抽象层
================================================================================

  层           调度对象              有标准吗            抢占性                公平性
  ---------------------------------------------------------------------------------------
  内核         可运行线程 → 核        有                 真抢占                设计目标即公平
                                     (EEVDF/CFS +        (时钟中断兜底)        (针对 CPU 时间)
                                      调度类 + 亲和性/
                                      NUMA)
  ---------------------------------------------------------------------------------------
  线程池       任务 → 空闲 worker    无统一标准          通常非抢占            取决于队列纪律
                                     (实现自定:          (run-to-completion)
                                      FIFO 全局队列 /
                                      work-stealing)
  ---------------------------------------------------------------------------------------
  GIL          等待者 → 下个持有者   只有薄机制          协作式 + 超时请求     不保证
                                     (5ms + drop          (边界才检查)
                                      request)，
                                      无获取策略

规律：越往用户态上层走，调度策略越稀薄、越没有强制标准，最后甚至把“选谁”
      的决定权又甩回给内核。


================================================================================
二、内核调度 OS 线程的标准
================================================================================

内核调度器只认 OS 线程。它眼里只有一堆处于“可运行 (TASK_RUNNING)”状态的
线程，以及一组核，然后决定谁上哪个核、跑多久。

1) 调度类（优先级从高到低）
   SCHED_DEADLINE > 实时类(SCHED_FIFO/SCHED_RR) > 普通类(SCHED_NORMAL)
   > batch > idle
   高优先级类里有可运行任务，低优先级类就拿不到核。绝大多数线程走普通类。

2) 普通类的核心逻辑
   - CFS（完全公平调度）：每个任务维护 vruntime，跑得越久涨得越多；nice 值
     越低（优先级越高）权重越大、vruntime 涨得越慢。调度器永远挑 vruntime
     最小的上核——谁亏欠得最多就补谁。
   - EEVDF（内核 6.6，2023 年底起替代 CFS）：引入 lag（滞后量）和虚拟截止
     期，在保证公平的同时更好照顾延迟敏感任务，挑“有资格且虚拟截止期最早”
     的任务。

3) 公平/优先级只决定“先后”，位置还要叠加几个约束
   - CPU 亲和性 (affinity)：sched_setaffinity / taskset 设的掩码，硬限制线程
     只能在某些核上跑（训练绑核动的就是这个）。
   - 缓存热度：线程被唤醒时倾向放回上次跑的核，L1/L2 还热，避免 cache miss。
   - NUMA 局部性：尽量让线程待在它内存所在 NUMA 节点对应的核上。
   - 负载均衡：周期性把可运行线程从忙核迁到闲核，但迁移有代价（缓存失效），
     带阻尼。
   - 抢占：更高优先级任务就绪、或时间片耗尽时，当前线程被抢占。

一句话：
   优先级/公平决定“先后”，亲和性/缓存/NUMA 决定“位置”，时间片/抢占决定“多久”。


================================================================================
三、线程池的任务调度（纯用户态）
================================================================================

线程池本质：程序启动时用 pthread_create / new Thread / std::thread 预先创建
好一批普通 OS 线程，让它们循环从任务队列取活干，省掉反复创建/销毁的开销。

★ 对内核来说，池里的线程和随手单独创建的线程没有任何区别。内核不知道“线程
  池”这个概念，只看到一个个 OS 线程（Linux 上即一个个 task，有 PID/TID，出现
  在 /proc、top -H、ps -T 里）。

1) 两大类队列纪律（无规范级统一标准，实现自定）
   - 单一全局队列 + FIFO（最朴素）：一个共享队列，worker 抢着取，先进先出。
     Java ThreadPoolExecutor、Python ThreadPoolExecutor 基本是这套。缺点：核
     多了所有 worker 抢同一把队列锁，竞争上升。
   - Work-stealing 工作窃取（高并行事实标准）：每个 worker 有自己的双端队列，
     自己从一端 push/pop（LIFO，局部性好），空闲时从别人队列另一端偷（FIFO，
     偷最老/最大的子树）。Java ForkJoinPool、Intel TBB、Cilk、Rust rayon/tokio、
     Go runtime 的 P-local queue 都属此家族。

2) 与内核调度最本质的区别：通常非抢占（run-to-completion）
   worker 取走一个任务就一口气跑完，池子没法像内核那样把任务从 worker 手里抢
   下来换一个。异步运行时（asyncio、tokio）是协作式的——任务只在 await 点让出。
   所以异步运行时里跑 CPU 密集任务会饿死整个 event loop：没有 await，调度器没
   有切换机会。内核有时钟中断兜底，用户态这层没有。

3) 线程池“竞争”分两层，别混
   - 任务队列层（用户态）：多个 worker 抢队列里的任务，自己用锁/无锁队列协调。
   - CPU 层（内核态）：这些 worker 作为 OS 线程，按内核标准抢核。
   池大小设成核数附近，本质是想让“可运行线程数 ≈ 核数”，减少上下文切换和负载
   均衡开销——但这只是用户态经验值，内核并不知道你“打算”这么干。

   （连到 PyTorch：intra-op 并行底层就是一个线程池(OpenMP 或 native)，
    torch.set_num_threads 控的就是它；inter-op 是另一个池。op 怎么分给线程是
    PyTorch/OpenMP 的策略，内核一无所知。）


================================================================================
四、Python GIL 的调度（纯用户态）
================================================================================

GIL = CPython 解释器内部一把全局锁，保证任意时刻只有一个线程在执行 Python
字节码。用一个 mutex + 条件变量实现。内核完全不知道 GIL 的存在。

★ 为什么 N 个 Python 线程在 N 个核上跑不出并行？
  因为没拿到 GIL 的线程根本不是“可运行”状态——它们阻塞在 GIL 的条件变量上睡
  着 (TASK_INTERRUPTIBLE)，压根不进调度器候选池。哪怕有 64 个空闲核、64 个
  Python 线程，内核看到的“可运行 Python 线程”始终只有 1 个，自然只占 1 个核。

1) GIL 切换机制的演变
   - 旧 GIL（3.2 之前）：按字节码条数计，跑满 N 条指令(sys.setcheckinterval，
     默认 100)就释放再重拿。多核上是 GIL 颠簸的元凶。
   - 新 GIL（3.2 起，Antoine Pitrou 重写）：改成基于时间。

2) 新 GIL 的共享状态（ceval_gil.c）
   - locked：GIL 当前是否被持有（原子）
   - mutex + cond：保护状态的互斥量，以及“GIL 释放时”唤醒等待者的条件变量
   - switch_number：每次易手 +1 的计数器
   - last_holder：上一个持有者
   - gil_drop_request：请求当前持有者让出的标志，挂在 eval breaker 上
   - interval：setswitchinterval 设的值，默认 5ms


================================================================================
五、新 GIL 的交接协议（详解）
================================================================================

★ 关键前提：5ms 这个超时属于“等待者”，不是属于“持有者”。
  不是“持有者跑满 5ms 就让出”，而是“等待者干等满 5ms 还没轮到，才去请求持有者
  让出”。持有者在被请求之前，完全不知道有人在等，它只管闷头跑字节码。

时序（线程 A = 持有者，线程 B = 等待者）：

  1. [B] take_gil：发现 locked==1 拿不到，去 cond 上带超时睡 (COND_TIMED_WAIT，
        超时值 = 5ms)。这 5ms 计的是“我作为等待者已经干等了多久”。
  2. [A] 同时在执行字节码，持有 GIL。
  3. [B] 5ms 超时醒来，判断：若超时醒 且 GIL 还被占 且 switch_number 跟睡前
        一样（说明同一个家伙一直占着、期间没易手）→ 置 gil_drop_request = 1，
        点亮 eval breaker。
  4. [A] 执行到下一个字节码检查点（eval breaker，指令边界，尤其循环回跳/函数
        调用处）→ 看到 drop 标志 → 执行 drop_gil：locked=0，COND_SIGNAL 唤醒
        一个等待者。
  5. [B] 被唤醒，重新锁 mutex、确认 locked==0，取得 GIL：locked=1，
        switch_number++，last_holder=B。
  6. [A] FORCE_SWITCHING：被 drop 请求逼着释放时，A 释放后会去 switch_cond 上
        等，直到 last_holder 不再是自己才继续——确保确实有别人接手了再走。

★ “两套独立节奏”：A 执行字节码的节奏 与 B 干等的 5ms，本来毫不相干；只有 B
  耐心耗尽去拍 A 肩膀（置 drop_request），两条时间线才被强行拉到一起握一次手。
  注意：当持有者时间片耗尽被内核抢占下核时，它并不释放 GIL，只是被挪下核，锁
  还攥在手里。


================================================================================
六、两个“但是”：GIL 既不公平，也不是真抢占
================================================================================

────────────────────────────────────────────────────────────────────────────
but 1：不保证公平 —— GIL 不决定“选谁”，把决定甩给了 OS
────────────────────────────────────────────────────────────────────────────

drop_gil 只做一件事：COND_SIGNAL(cond)，唤醒一个等待者。若同时有 B、C、D 三个
线程在等，到底哪个被唤醒、抢到 mutex 进而拿到 GIL，完全由 OS 的 condvar/mutex
唤醒语义决定。pthread 条件变量不保证 FIFO，也不认 nice 值优先级。

★ FORCE_SWITCHING 的补丁（修正一个常见的过强说法）：
  新 GIL 加了 FORCE_SWITCHING 机制。当持有者是被 drop 请求逼着释放的，它释放后
  会等到 last_holder 变成别人才继续，所以“刚放手又被同一个线程立刻抢回去”这个
  最坏情况在“有 drop 请求时”是被缓解的。但要点不变：
   - 只在“被请求让出”时生效；主动释放（去做 I/O、调用主动放 GIL 的 C 扩展）时
     没有这套保护。
   - 即便有 FORCE_SWITCHING，“接手的是 B 还是 C”依然 OS 说了算，多个等待者之间
     没有公平序，饿死某个线程仍可能。
   - 多核历史上的 GIL convoy 颠簸，本质就是“GIL 释放/唤醒时机”和“内核在核间调
     度的时机”对不齐，大量时间耗在反复唤醒-抢锁-切换上。

────────────────────────────────────────────────────────────────────────────
but 2：协作式，不是真抢占 —— 5ms 是“请求”的节奏，不是“保证”的上限
────────────────────────────────────────────────────────────────────────────

gil_drop_request 被置上后，不会主动打断 A。它只是静静挂在 eval breaker 上，等
A 自己执行到下一个字节码检查点时顺手瞄一眼才被发现。

只要 A 老实跑字节码，很快撞到检查点，5ms 后顺利交接——正常。
问题出在 A 卡进一条不回解释器循环的长操作里：
   - 灾难性回溯的正则：CPython 的 re 匹配在 C 层一气跑完，中途不回解释器循环，
     期间也不释放 GIL。匹配几秒，这几秒里 eval breaker 一次都不检查。
   - 不主动放 GIL 的 C 扩展：某 C 扩展做耗时纯 C 计算，却没用
     Py_BEGIN_ALLOW_THREADS / Py_END_ALLOW_THREADS 放掉 GIL，则全程攥着锁。
     （反例：numpy/PyTorch 恰恰在重计算前主动放 GIL，所以不卡别的线程。）

★ 精确含义：5ms 只决定 B “多久之后开口请求让出”，并不决定 A “多久之内一定让出”。
  A 何时真正让出取决于它何时回到字节码边界；卡在长 C 操作里时，实际持锁可远超
  5ms，谁也抢不走——没有任何东西能像内核时钟中断那样强行打断它。

共性：内核那层有时钟中断兜底，是带强制力的真抢占；GIL 这层既不强制（只能请求、
      等对方自觉到检查点），也不公平（选谁交还给 OS）。


================================================================================
七、“谁拿到 GIL”是两次独立选择串起来的结果
================================================================================

常见误解：“内核调度 Python 线程的执行，但没有公平策略”。这句是错位的——

★ GIL 的等待者根本不在内核视野里：B/C/D 都阻塞在 cond 上睡着 (TASK_INTERRUPTIBLE)，
  不进调度器候选池。内核看不见睡着的线程，谈不上对它们公平不公平。内核的公平
  (EEVDF/CFS) 只作用于已经可运行的线程，管的是“醒着的线程怎么分 CPU 时间”。

“谁拿到 GIL” = 三步串联，没有任何一步是为“GIL 公平”设计的：

  第一次选择 —— 唤醒谁（condvar 干的，不是调度器）
     A 释放时 COND_SIGNAL 只唤醒一个等待者。唤醒哪个由 pthread 条件变量实现决定，
     POSIX 不作保证：不保证 FIFO、不认 nice 值。就这一步把 B/C/D 里某一个（假设
     C）从睡眠态变回可运行。选的是“谁醒”，决定权在 condvar，跟内核调度器无关。

  第二次选择 —— 可运行之后谁先上核（这才是内核干的，且公平）
     C 变成可运行，内核才开始管它，按 EEVDF 排进某个核去跑。但内核公平的是
     “CPU 时间”，不是“GIL 归属”；此刻通常只有 C 一个被唤醒者，内核也没在
     B/C/D 之间选，只是决定 C 何时上核。

  最后一道坎 —— 抢 mutex
     C 上核后要重新锁 mutex、确认 locked==0 才拿走 GIL。中间可能杀出别的线程：
     比如刚做完 I/O 回来、或主动放过 GIL 现在要重新拿的线程，它根本不在 B/C/D 的
     等待队列里，却可能抢先锁到 mutex 把 GIL 叼走（插队 / stealing）。C 即使被
     唤醒也不保证赢。

★ 症结：内核的“CPU 时间公平” ≠ “GIL 获取公平”。一个线程完全可以被内核公平调度、
  拿到应得的 CPU 时间片，却在 GIL 这件事上一次次抢输、长期饿死——因为没有任何
  一层负责“等得久的等待者优先拿到 GIL”。GIL 的公平性从 condvar 和调度器的缝隙
  里漏掉了，谁都没接这个责任。


================================================================================
八、COND_SIGNAL“唤醒哪一个”是随机吗？
================================================================================

更准确：未指定 (unspecified)，不是“刻意随机”。
   - POSIX 只承诺“唤醒至少一个等待者”，不规定是哪一个，选择权完全留给实现。
   - 不是用随机数抽签，所以不是真随机。
   - 取决于实现的内部数据结构：等待者在内核里挂在 futex 的 wait queue 上，
     signal 唤醒哪个取决于队列组织方式和唤醒策略。
   - Linux glibc 新实现（基于 futex）大体接近 FIFO，但不是规范承诺，不能依赖；
     换平台/换 libc 版本/掺入超时唤醒或虚假唤醒(spurious wakeup)就可能变样。

★ 工程心智：当作“无序、不可预测”对待——可能近似 FIFO，但写代码时一丁点都不能
  假设顺序或公平。


================================================================================
九、线程池线程 vs GIL 等待者：机制同构
================================================================================

“线程池里的线程对内核可见吗？是运行的吗？拿到任务还能被内核发现吗？”

1) 可见——完全是内核一等公民。线程池不是特殊内核实体，就是一批普通 OS 线程。

2) “是不是运行”取决于这一刻的状态：
   - 正在跑某个任务（CPU 计算）→ 可运行/运行态 (TASK_RUNNING) → 内核按 EEVDF
     排到核上跑。
   - 空闲、阻塞在任务队列上等活（队列空，睡在条件变量/信号量上）→ 睡眠态 →
     不在调度候选池，不占核。

   ★ 这跟 GIL 等待者是同一个机制：空闲池线程 和 等 GIL 的 Python 线程，都阻塞
     在某个同步原语上睡着、对调度器隐身；被唤醒才变回可运行、重进内核视野。
     区别只在等什么：池线程等“任务队列来活”，GIL 等待者等“GIL 空出来”。

3) “拿到任务还能被内核发现？”——因果反了：
   不是“内核去发现某线程拿到任务”，而是“线程拿到任务这件事本身就把它变成可运行，
   内核自然就调度它”。完整链条（和 GIL 那条一模一样）：
     ① 池线程空闲 → 阻塞在任务队列条件变量上睡 → 内核看不见。
     ② 提交任务塞进队列，signal 条件变量 → 唤醒一个池线程（唤醒哪个 = 第八节
        那套未指定语义）。
     ③ 被唤醒线程从睡眠态变回可运行态 → 这一步就重新进入调度候选池。
     ④ 内核按 EEVDF + 亲和性/NUMA/缓存热度，决定它何时、上哪个核跑任务。
     ⑤ 任务跑完，回队列等下一个 → 又睡下 → 又从内核视野隐身。

   ★ “被内核发现”不是额外动作，而是状态变化的自动结果：线程从“睡眠”翻转成
     “可运行”的那一刻，就自动回到调度器候选集合。内核不需要去“发现谁拿了任务”，
     它只认一件事——你现在是不是 TASK_RUNNING。是就参与抢核，不是就当你不存在。


================================================================================
十、收束
================================================================================

内核眼里从来只有“OS 线程 + 它此刻的状态”这两样。线程池、任务队列、GIL，全是
用户态在上面叠的协调逻辑，内核一概不知；它们能影响调度的唯一方式，就是通过
“睡眠 ↔ 唤醒”改变线程的可运行状态：
   - 线程池靠“任务到达”来唤醒；
   - GIL 靠“锁释放”来唤醒；
   - 机制完全同构，而“唤醒谁”那一步又都落在同一套未指定的 condvar 语义上。

补充趋势：CPython 3.13 起提供实验性 free-threaded 构建 (PEP 703, --disable-gil)，
去掉 GIL 后“GIL 调度”这一层直接消失，Python 线程退化成普通 OS 线程，整个问题
只剩“内核调度 + 自己用锁保护共享状态”两层，与 C++/Java 模型对齐。代价是 C 扩展
兼容性和单线程性能，目前仍在逐步成熟。
