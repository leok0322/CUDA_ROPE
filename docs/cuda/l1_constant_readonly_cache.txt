================================================================================
GPU 片上缓存：L1 data cache / constant cache / 只读数据缓存(__ldg)
——三者区别、各自适用场景与代码示例
================================================================================

适用：NVIDIA GPU。结论以 Maxwell 及以后(尤其 Ampere sm_86，本机)为准，
      老架构(Kepler)的差异单独标注。


================================================================================
一、先放结论(一句话区分)
================================================================================

  · L1 data cache    ：每 SM 的【通用】数据缓存，缓存普通全局/局部内存访问；最常用、最通用。
  · constant cache   ：每 SM 的【小】缓存，只服务 __constant__ 常量空间；warp 内【同址广播】最快，
                       地址发散就【串行】。
  · 只读数据缓存(__ldg)：缓存【全局内存】的【只读】访问(ld.global.nc 非一致加载)。Kepler 时是
                       一块【独立】的 48KB 纹理/只读缓存；Maxwell 起【与 L1 物理统一】——此后
                       __ldg 的主要价值不再是"额外缓存"，而是给编译器【只读+不别名】的提示。

  内存层次(由近到远)：寄存器 → L1/shared(每 SM 片上) → L2(全设备共享) → HBM(显存)。
  constant cache 与只读缓存都属"L1 同层"的特化通路；shared memory 是【可编程便笺】，
  与 L1 共用同一块 SRAM(可配置切分)，但不是"缓存"，不在本文比较之列。


================================================================================
二、L1 data cache（通用数据缓存）
================================================================================

  · 位置/范围：每个 SM 私有；与 shared memory 共用一块片上 SRAM(carveout 可配，
    如 sm_86 合计 128KB/SM，在 L1 与 shared 间分配)。
  · 缓存什么：普通全局内存(global)与局部内存(local/寄存器溢出)的 load/store。
  · 默认行为：Volta(7.0)及以后，普通全局 load【默认就缓存在 L1】(+L2)；
    Kepler 时全局 load 默认只走 L2、绕过 L1(需 -Xptxas -dlcm=ca 才进 L1)。
  · 一致性：L1【非跨 SM 一致】——别的 SM/线程写的数据，本 SM 的 L1 不会自动失效；
    所以编译器对"可能被写"的数据会保守(见第四节别名问题)。
  · 缓存策略(PTX cache operator，可经内联汇编/编译选项控制)：
      .ca (cache all：L1+L2)   .cg (cache global：只 L2、绕 L1)
      .cs (streaming：用完快逐出，少污染)   .cv (volatile：每次重读)
    （LayerNorm 项目里用 .cs hint 让一次性数据快速逐出 L1，就是这一类。）

  典型用法：什么都不用写——普通访问即走 L1：
      float v = a[i];            // 全局 load，Volta+ 默认缓存 L1+L2
      a[i] = v * 2.0f;           // 全局 store

  适合：通用的、可能复用、有空间局部性的全局数据(矩阵分块、邻域访问等)。


================================================================================
三、constant cache（常量缓存）
================================================================================

  · 服务对象：__constant__ 声明的【常量内存】(整设备共 64KB 常量空间)，以及 kernel 实参
    (编译器放进常量 bank)。普通全局指针【不】走它。
  · 缓存：每 SM 一块【很小】的常量缓存(约 8KB 量级)，独立于 L1。
  · ★最优访问模式 = warp 内【所有线程读同一地址】(uniform/broadcast)：
      一次广播即可服务整个 warp，极快(近 1 拍)。
  · ★最差：warp 内线程读【不同地址】(发散) → 硬件【串行化】，读 N 个不同地址花 ~N 倍。
  · 容量小、且数据须事先放进常量空间(host 端 cudaMemcpyToSymbol 或编译期常量)。

  用法：
      __constant__ float c_coef[256];               // 设备端常量数组
      // host：cudaMemcpyToSymbol(c_coef, h_coef, sizeof(float)*256);
      __global__ void k(...) {
          float w = c_coef[blockIdx.x];   // ★好：同一 warp 内 blockIdx.x 相同 → 同址广播
          float w2 = c_coef[threadIdx.x]; // ✗ 差：每 lane 不同地址 → 串行
      }

  适合：所有线程共用的少量只读标量/小表(超参、滤波器系数、查找表)，且访问是 warp 同址广播。
  不适合：大数据、或 warp 内地址发散的访问。

  ── __constant__ 变量的作用域 / 生命周期 / 设备维度（易混淆，单列）────────────────
   · 声明位置：必须在【函数外】(文件/命名空间作用域)，不能在 kernel 内部声明；不能动态分配、
     大小编译期固定；整设备常量空间总量 ≤ 64KB(所有 __constant__ 共享)。
   · 设备端【只读】，host 端用 cudaMemcpyToSymbol / cudaMemcpyFromSymbol 读写：
       cudaMemcpyToSymbol(c_coef, h_coef, sizeof(float)*256);   // 不能用普通 cudaMemcpy(&c_coef,..)
   · 生命周期：随 CUDA context 全程存在，【跨多次 kernel 启动保持】，不是每个 kernel 重新初始化。
   · ★"谁能访问"是【模块(module)】层面，不是【显卡(device)】层面——两条正交维度：
       - 模块 = 编译/链接到一起的设备代码镜像(CUmodule，软件概念)。同一模块里【所有 kernel、
         所有 block/线程】都能按名引用该 __constant__。
         · 默认(whole-program, -rdc=false)：每个 .cu 各一份，跨 .cu 不共享；
         · 跨 .cu 共享需 `extern __constant__` + 可重定位设备代码(-rdc=true / SEPARABLE_COMPILATION)。
       - 设备 = 物理 GPU。【每块 GPU 各有该常量的独立物理副本】；cudaMemcpyToSymbol 写的是
         【当前 cudaSetDevice 选中的那块卡】上的那一份，多卡需逐卡设置：
             cudaSetDevice(0); cudaMemcpyToSymbol(c_coef, h0, ...);  // 设备0 的副本
             cudaSetDevice(1); cudaMemcpyToSymbol(c_coef, h1, ...);  // 设备1 的副本(互不影响)
     ⟹ 模块决定"哪些 kernel 编译期能引用它"；设备决定"用哪块卡上的那份物理副本"。


================================================================================
四、只读数据缓存（__ldg / ld.global.nc）
================================================================================

  · 来源：对【全局内存】的只读访问，经"只读/纹理"通路加载，PTX 为 ld.global.nc
    (nc = non-coherent，非一致：承诺本 kernel 期间该数据不被写)。
  · 历史：Kepler(sm_35) 上是一块【独立的 48KB 只读/纹理缓存】，和 L1 分开 → 当年能"白捡"
    一块额外缓存容量，收益明显。Maxwell(5.x)起【L1 与只读/纹理缓存物理统一】成一块 SRAM。

  ★在统一缓存架构(Maxwell+/Ampere)上，__ldg 的真正价值【不是】"额外开一个缓存"
   (普通 load 本来也进 L1、也能复用命中)，而是给【编译器/硬件】两个承诺：
     ① 只读 + 不别名：本 kernel 里【有写】别的全局内存(如就地改写 qkv)时，编译器一般无法
        证明只读指针与那些 store 不重叠 → 保守起见，store 之后可能【重读】该数据、不敢把它
        留在寄存器跨 store 复用。__ldg(或 const __restrict__) 声明只读不别名 → 编译器可
        【把值留寄存器、跨 store 复用、不再重发 load】。
     ② 非一致加载：不必担心被本 kernel 的 store 影响，缓存行处理更宽松。
   ⟹ 收益是"少发冗余 load 指令 / 寄存器复用"，而非"让缓存命中"。在 Ampere 上对小而只读的
      数据，__ldg vs 普通 load 差距通常【很小】；它更像"零成本的正确提示" + "向后兼容老架构"。

  ── 详解 ①：指针别名(aliasing)如何逼出"重读"，__ldg 如何消除 ──────────────────────
   别名规则：编译器遇到 `load A → store B → 再用 A`，只要【证明不了 A、B 不重叠】，就必须假设
   "store B 可能改了 *A" → 不敢把第一次读到的 *A 留寄存器跨 store 复用，store 后【被迫重读 *A】。

   ★触发的【必要结构】是"读 X → store(可能别名) → 【再读】同一 X"——三步缺一不可，关键是
     store 之后【还要再用 X】：

       // 无 __ldg / 无 __restrict__，且 store 后再读同一地址：
       float c   = cos_ptr[i];      // load1
       qkv_ptr[j] = c * x;          // store 到 qkv（编译器证明不了 qkv 与 cos 不重叠）
       float c2  = cos_ptr[i];      // ★再读同一 cos_ptr[i] → 怕被上面 store 改 → 【被迫重读】

   __ldg(&cos_ptr[i])（或把参数声明 const __restrict__）= 承诺"只读、不被任何 store 改、不与
   store 别名" → 编译器可【把 c 留寄存器跨 store 复用，不再发 c2 的重读 load】；循环里反复"读→
   store→再读"时省下大量冗余 load。这就是 ① 的全部——【不是让它进缓存】，而是【消别名、免重读】。

   ★但要注意"再读"这一步是【必要条件】：若 store 之后【不再读】那块只读数据(它已是死值)，
     就【没有重读、也没有可省的 load】，__ldg 在此【省不了任何指令】。

  ── 本 kernel 实情：是"先读完 cos/sin、再统一 store qkv"，不触发重读 ──────────────────
   fused QK-Norm+RoPE 的结构是：
     循环1(RoPE)：读 cos/sin → 算出 elementPerThread(local)；cos/sin 用完即死，且每轮读不同地址。
     循环2(写回)：读 local → 末尾【一次】store 到 qkv；store【之后不再读 cos/sin】。
   ⟹ store 在所有 cos/sin 读【之后】，且之后无"再读" → 不构成"读→store→再读"，
      编译器【不会插入 cos/sin 的重读 load】。所以这里【没有可消除的重读】，__ldg(VLLM_LDG)
      省不了 load；它残留的只是 ② 那点很小的只读/非一致提示(Ampere 上近乎可忽略)，以及兼容老卡。
   （即"读→store→再读"才需要 __ldg；本 kernel 是"读完再 store"，不踩这个坑。）

  ── 详解 ②：非一致加载(ld.global.nc)——缓存行不必维护一致性、不被 store 失效 ──────────
   先分清 ① 和 ② 的层次（同一个"只读"承诺的两面）：
     · ① 是【编译器/寄存器】层面：把值留寄存器、跨 store 复用、不重发 load(上一条)。
     · ② 是【硬件/缓存行】层面：当值【不在寄存器】、确实要去缓存取时(被换出、或别的 lane 取)，
       那条缓存行怎么处理。

   "一致性(coherent)"指什么：L1 是【每 SM 私有、跨 SM 不自动一致】的。普通 cached load 取来的行，
   硬件要按内存模型【保守对待与 store 的关系】——若该地址【可能被写】，行可能要【失效/重取】以保证
   后续读能看到写的结果(维护一致性/可见性有开销)。

   ld.global.nc(nc=non-coherent) 的承诺："这块本 kernel 期间【只读、不会被任何 store 改】" ⟹
     · 硬件【无需为这条行维护一致性】，也【不会因 kernel 内的 store 去失效它】；
     · 行可以稳稳【常驻只读通路】、自由服务命中，不被写流量打扰；
     · 跨 SM 一致也成了非问题(反正不会变)。
   → 即"缓存行处理更宽松"：省掉一致性维护/失效判断，命中更稳。

   与 ① 配合：① 让能用寄存器时不去碰缓存；② 让真去缓存时那条行也不被 store 搅动。两者都源于
   "只读不别名"这一个承诺。

   ★前提(和 ① 一样)：数据必须【确实全程只读】。若你撒谎(其实有线程写它)，nc 缓存的副本可能【读到
     旧值】→ 未定义行为。这是程序员的保证(等同 const/__restrict__ 的正确性义务)。

   幅度：统一缓存架构(Maxwell+/Ampere)上普通 load 本就进 L1，故 ② 相对默认 load 的差别【通常很小】
     (主要是省一致性/失效处理)；它在【Kepler】上意义更大——那时只读缓存是【独立 48KB】，nc 加载
     等于多一块容量与带宽。

  ── 易混淆点：跨 warp 复用命中 L1，和 __ldg 无关 ────────────────────────────────────
   "同一 token 的多个 head(多个 warp)读同一行 cos/sin → 命中 L1 复用"这件事，【用不用 __ldg
    都发生】(靠 L1，普通 load 也缓存)。__ldg 的额外贡献只在【单线程内因别名被迫重读】这一层。

  ── 返回值：__ldg(arg) 返回【解引用后的值】，不是指针 ────────────────────────────
   签名： template<typename T> __device__ T __ldg(const T* address);
          收【指针】，返回【pointee 的值 T】——本质是一条 load 指令(把值读进寄存器)，故结果是值。
       const float* p;
       float v = __ldg(p);        // ✓ v = *p 的值(float)，经只读缓存；等价 *p，只是缓存通路不同
       // float* q = __ldg(p);    // ✗ 错：__ldg 不返回指针
   ⟹ 这也是 VLLM_LDG 两分支能互换的前提：__ldg(arg) 与 *(arg) 结果【都是 *arg 的值】，只是
      前者走 ld.nc(只读缓存)、后者普通解引用。

  用法（两种等价写法）：
      float v = __ldg(&a[i]);                       // 显式 intrinsic：返回 a[i] 的【值】
      // 或：参数声明只读受限指针，普通访问即自动用 ld.nc：
      __global__ void k(const float* __restrict__ a, float* out) {
          out[i] = a[i] * 2.0f;   // a 只读不别名 → 编译器可走 ld.nc / 寄存器复用
      }

  要求：被读数据在 kernel 执行期间【确实只读】(无线程写它)，否则行为未定义。
  适合：只读输入、且(尤其)同一 kernel 内还有写操作、需跨 store 复用只读值的场景。

五、三者对比表
================================================================================

  ┌──────────────┬───────────────┬────────────────┬─────────────────────────┐
  │              │ L1 data cache │ constant cache │ 只读数据缓存(__ldg)      │
  ├──────────────┼───────────────┼────────────────┼─────────────────────────┤
  │ 缓存的空间    │ global/local  │ __constant__   │ global(只读，ld.nc)     │
  │ 触发方式      │ 普通访问(默认)│ 访问__constant__│ __ldg() 或 const __restrict__ │
  │ 容量          │ 大(与shared共享SRAM) │ 很小(~8KB/SM) │ Kepler独立48KB；Maxwell+与L1统一 │
  │ 最优访问      │ 局部性好即可  │ warp内【同址广播】│ 只读、可发散但有局部性 │
  │ 发散地址      │ 正常按line命中│ ★串行化(慢)    │ 正常命中                │
  │ 跨SM一致      │ 否            │ 否(只读)        │ 否(non-coherent，假定只读)│
  │ 是否需声明    │ 否            │ 须 __constant__ │ 否(用__ldg/__restrict__)│
  │ 现代架构收益  │ 基线          │ 广播场景大      │ 主要是【只读+不别名提示】，缓存层面≈L1 │
  └──────────────┴───────────────┴────────────────┴─────────────────────────┘


================================================================================
六、怎么选（决策）
================================================================================

  问：数据是 warp 内【所有线程读同一个值】的少量只读量(超参/小系数表)？
      → 用 __constant__（广播极快）。注意发散访问会串行。

  问：只读全局数据，且 kernel 内还【写】别的全局内存(如就地改写)？
      → 用 __ldg / const __restrict__（消除与写的别名顾虑，寄存器复用、少重读）。

  问：一般的全局数据(可读可写、或读多但无上面两特征)？
      → 直接普通访问，走 L1 即可，别画蛇添足。

  问：一次性流式、读完不再用？
      → 普通访问 + 可选 .cs streaming hint，减少 L1 污染(见 LayerNorm 项目 K9)。


================================================================================
七、本项目落点（fused QK-Norm + RoPE）
================================================================================

  · qkv：可读【可写】(就地改写) → 普通访问走 L1；用向量化+合并保证带宽(见
    docs/cuda/SMEM_Bank_Conflict_and_GMEM_coalesce.txt)。
  · cos/sin：只读 RoPE cache，且【同一 token 的 Hq+Hk 个 head(多个 warp)读同一行】、
    复用度高 → 用 VLLM_LDG(=__ldg) 加载(src/inference/fused_qknorm_rope_kernel.cu)。
      - 复用命中其实靠 L1 即可(普通 load 在 Ampere 也缓存)；__ldg 的额外价值是：cos/sin 只读，
        而 kernel 同时在写 qkv，__ldg 让编译器确信二者不别名 → cos/sin 可寄存器复用、不被
        qkv 的 store 逼着重读。在 sm_86 上这点收益不大，属"零成本提示 + vLLM 惯例 + 兼容老卡"。
  · eps/标量超参：少量、warp 同址 → 适合常量/寄存器(若放 __constant__ 则走常量缓存广播)。

  ★澄清(与早前讨论一致)：cos/sin 不走 constant cache——它在全局内存、且 warp 内各 lane 读
   不同下标(发散)，constant cache 发散会串行；只读数据缓存/L1 才合适。
