================================================================================
Kernel 0（base）BLOCK_SIZE Autotune 分析（sm_86 / RTX 3090）
================================================================================


一、Autotune 方法
--------------------------------------------------------------------------------

    搜索空间：BS ∈ {32, 64, 128, 256, 512, 1024}（均为 32 的倍数，覆盖完整 warp）
    测试规模：m ∈ {128,256,512,1024,2048,4096}，n ∈ {128,256,512,1024,2048,4096}，共 36 组
    评估指标：每组 size 的平均 GFLOPS（多次重复取均值）


二、测试结果汇总
--------------------------------------------------------------------------------

    BLOCK_SIZE   平均 GFLOPS   峰值 GFLOPS   最低 GFLOPS
    ──────────   ───────────   ───────────   ───────────
    32              63.0          92.8          2.6
    64              69.0          96.5          2.9
    128             73.6         102.6          3.8   ← 最优
    256             65.8         101.1          1.9
    512             62.6          97.8          3.4
    1024            62.0          90.3          3.4

    最优 BLOCK_SIZE = 128（平均 GFLOPS 最高，峰值最高）


三、sm_86 硬件约束与占用率
--------------------------------------------------------------------------------

    sm_86（RTX 3090）关键参数：
        max threads/SM  = 1536
        max blocks/SM   = 16
        kernel_base 寄存器用量 = 18 个/线程（ptxas 输出）

    各 BS 的 SM 占用率（寄存器不是瓶颈：65536/(18×128)=28 blocks）：

    BS     blocks/SM（线程限制）       实际 blocks/SM   threads/SM   占用率
    ────   ────────────────────────   ──────────────   ──────────   ──────
    32     1536/32=48 → 上限 16            16              512        33%
    64     1536/64=24 → 上限 16            16             1024        67%
    128    1536/128=12（< 16）              12             1536       100%  ← 满占用
    256    1536/256=6                        6             1536       100%
    512    1536/512=3                        3             1536       100%
    1024   1536/1024=1                       1             1024        67%


四、为什么 BS=128 最优
--------------------------------------------------------------------------------

4.1 BS=32/64 差的原因：max_blocks/SM 截断，全局活跃 warp 严重不足

    bs=32：每 SM 驻留 16 blocks × 32 threads = 512 threads = 16 warp（33% 占用）
    bs=64：每 SM 驻留 16 blocks × 64 threads = 1024 threads = 32 warp（67% 占用）

    SM 有容纳更多线程的空间，但 block 数上限（16）阻止了继续填入。
    latency hiding 发生在 SM 内部，warp 调度器只能在该 SM 的驻留 warp 间切换。
    当 SM 内所有 warp 都在等内存时，SM 空转——多个 SM 并行运行也无法弥补。

    workload 大时所有 SM 全忙，全局活跃 warp 总数：
        BS=32：82 SM × 16 warp = 1312 warp
        BS=128：82 SM × 48 warp = 3936 warp（3× 差距）

4.2 BS=1024 差的原因：block 太大，剩余线程槽装不下第二个 block

    放入 1 个 block 后，剩余 1536-1024=512 个线程槽，
    不够再放一个完整的 block（需要 1024）→ 512 槽浪费 → 67% 占用。

    小 BS 可以一批批继续填入（BS=128 可塞 12 个），直到填满 1536 线程槽。

    BS=1024 的性能损失主要体现在中等尺寸：
        block 总数够覆盖所有 SM，但每 SM 只有 32 warp（vs BS=128 的 48 warp）。
        实测 (512,512)：BS=128 得 93.2G，BS=1024 得 51.9G（差距 ~80%）。

    小尺寸下：BS=1024 与 BS=128 的全局 warp 总数相同（更少 SM 但每 SM warp 更多），
    性能接近，差异为测量噪声。

4.3 BS=128 是最优甜点

    满足 100% 占用的 blockSize 条件（两个约束均不成为瓶颈）：
        ceil(max_threads_SM / max_blocks_SM) ≤ blockSize ≤ max_threads_block
              ceil(1536 / 16) = 96           ≤ blockSize ≤ 1024

    sm_86 上满足条件的 2 的幂次：128、256、512，三者均达到 100% 占用。

    BS=128 是其中最小的，相同工作量下启动的总 block 数最多（BS=256 的 2 倍），
    中小尺寸下能让更多 SM 同时参与计算。

    注意：BS=128/256/512 每 SM 的 warp 数相同（均为 48），
    warp 调度器行为一致，block 数本身不影响调度性能。
    三者在大尺寸（SM 全忙、带宽饱和）下性能趋于收敛（90~102 GFLOPS）。


五、结论
--------------------------------------------------------------------------------

    最优 BLOCK_SIZE = 128

    核心原因：
        1. 满足 100% SM 占用（1536 线程/SM，48 warp/SM）
        2. 是达到满占用的最小 2 的幂次，总 block 数最多，对中小尺寸最友好
        3. BS=32/64 被 max_blocks=16 截断，全局活跃 warp 只有 BS=128 的 1/3 或 2/3
        4. BS=1024 block 太大，SM 内剩余槽位不足以再塞一个完整 block，占用率 67%

    适用范围：
        对 sm_86 上所有寄存器用量低（≤ ~42 个）、不使用 shared memory 的 kernel 均成立。
        若寄存器或 shared memory 用量较高，实际瓶颈会转移，需重新 autotune。
