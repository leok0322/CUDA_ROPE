================================================================================
函数的变参：C 风格 ...（va_list） 与 C++ 变参模板，并与宏变参 __VA_ARGS__ 对比
================================================================================

"接收任意多个参数"不是宏独有的——函数也有两条路。本文讲【C 风格变参】和【C++ 变参模板】，
并和宏的 __VA_ARGS__ 对比（核心差异：宏传【源码文本】，函数传【求值后的值】）。

  目录：
    一、三种"变参"总览
    二、C 风格变参：... + <cstdarg>（va_list 那套）
    三、C++ 变参模板：template<typename... Args>（形参包 + 折叠表达式）
    三·补、实战实例：c10::str 与 torchCheckMsgImpl 的实现 + 调用追踪
    四、与宏变参 __VA_ARGS__ 的对比（传文本 vs 传值）
    五、怎么选
    六、小结


================================================================================
一、三种"变参"总览
================================================================================

  - 宏变参      ：#define F(fmt, ...)  里的 ... / __VA_ARGS__ —— 预处理层，传【源码文本】。
  - C 风格变参  ：int printf(const char* fmt, ...); —— 运行期从栈/寄存器按约定取值，无类型安全。
  - C++ 变参模板：template<typename... Args> void f(Args... args); —— 编译期实例化，有类型、安全。

  典型代表：TORCH_CHECK(宏变参)、printf(C 变参)、c10::str(C++ 变参模板)。


================================================================================
二、C 风格变参：... + <cstdarg>（va_list 那套）
================================================================================

  形参表末尾写 ...，函数体内用 <cstdarg> 的宏遍历这些"没名字的参数"：
      va_list ap ：保存遍历状态的对象
      va_start(ap, last) ：初始化，last 是 ... 前面【最后一个具名参数】
      va_arg(ap, T)      ：取下一个参数，【你必须告诉它类型 T】
      va_end(ap)         ：收尾

  例：求和（调用方必须先传"有几个"）
      #include <cstdarg>
      int sum(int count, ...) {
          va_list ap;
          va_start(ap, count);          // count 是 ... 前最后一个具名参数
          int s = 0;
          for (int i = 0; i < count; ++i)
              s += va_arg(ap, int);     // 逐个取，按 int 解释
          va_end(ap);
          return s;
      }
      sum(3, 10, 20, 30);   // → 60

  ★ 致命缺点：【没有类型安全】
    - 编译器不知道你到底传了几个、什么类型；全靠"约定"(如 printf 的格式串、或第一个 count 参数)。
    - va_arg(ap, T) 的 T 写错 / 个数对不上 → 未定义行为(读错内存)。printf("%d", 3.0) 就是这类坑。
    - 实参还会经历"默认实参提升"(char/short→int、float→double)，更易和你以为的类型对不上。
  ⟹ C 变参是【运行期】按调用约定取值，无编译期检查；能用 C++ 变参模板就别用它。


================================================================================
三、C++ 变参模板：template<typename... Args>（形参包 + 折叠表达式）
================================================================================

  用【形参包(parameter pack)】表示"任意多个、各自有类型"的参数：
      template <typename... Args>     // Args 是【类型】包
      void f(const Args&... args);    // args 是【值】包(这里按 const 引用，避免拷贝)

  - sizeof...(args) / sizeof...(Args) ：取参数【个数】(编译期常量)。
  - 展开形参包：常用 C++17【折叠表达式(fold expression)】，把一个运算符套在整个包上：
        template <typename... Args>
        void print(const Args&... args) {
            (std::cout << ... << args) << '\n';   // 折叠：((cout<<a0)<<a1)<<a2 ...
        }
        print("x=", 3, " y=", 4.5);               // 输出: x=3 y=4.5
    ── 折叠表达式的【四种形式】（设包 args = e1,e2,…,en，op 是二元运算符）──────────────
      写法                       名称          展开成                       结合方向
      -------------------------  ------------  ---------------------------  --------
      ( pack op ... )            一元右折叠     e1 op (e2 op (… op en))      右结合
      ( ... op pack )            一元左折叠     ((e1 op e2) op …) op en      左结合
      ( pack op ... op init )    二元右折叠     e1 op (… op (en op init))    右结合
      ( init op ... op pack )    二元左折叠     ((init op e1) op …) op en    左结合
      记忆：【... 在 pack 右边=右折叠、左边=左折叠】；【init 永远紧挨 ...】(在 pack 另一侧)。
      ★ 折叠必须带外层圆括号 ( ... )；op 可用 + - * / % ^ & | << >> && || , == 等约 32 个运算符。

    ── 四种形式【各举一例】（统一用 - 和输入 1,2,3，便于看清结合差异）──────────────
      // ① 一元右折叠 ( pack op ... )
      template<class... A> int f1(A... a){ return (a - ...); }
      f1(1, 2, 3);   // 展开 1-(2-3)        = 1-(-1) =  2
      // ② 一元左折叠 ( ... op pack )
      template<class... A> int f2(A... a){ return (... - a); }
      f2(1, 2, 3);   // 展开 (1-2)-3                 = -4
      // ③ 二元右折叠 ( pack op ... op init )，init=100
      template<class... A> int f3(A... a){ return (a - ... - 100); }
      f3(1, 2, 3);   // 展开 1-(2-(3-100))           = -98 ； f3() 空包 → 100(=init)
      // ④ 二元左折叠 ( init op ... op pack )，init=100
      template<class... A> int f4(A... a){ return (100 - ... - a); }
      f4(1, 2, 3);   // 展开 ((100-1)-2)-3           = 94  ； f4() 空包 → 100(=init)
      ⟹ 对比可见：左/右折叠【结合分组不同】(对 - 结果不同：①右=2、②左=-4)；二元版多一个 init，
        且【空包时结果=init】(一元版空包对 - 直接编译错)。

    ── 各形式的【常见惯用】（实战里一般这么用）─────────────────────────────────────
      一元右折叠 (a && ...)            // 是否全为真；&& 是空包合法的少数例外(空→true)
      一元左折叠 (... + a)             // 求和(无 init；空包对 + 非法)
      二元左折叠 (std::cout << ... << a)  // 顺序打印(init=cout) —— 本节 print 用的就是它
      二元左折叠 (0 + ... + a)         // 求和(init=0；空包安全→0)，比一元版能处理 0 参

    ── 左/右折叠对【顺序敏感】的运算符结果不同(用 - 看清)──────────────────────────
        (a - ...)    // 右折叠：1 - (2 - 3) =  2
        (... - a)    // 左折叠：(1 - 2) - 3 = -4
      加法 (a + ...) 与 (... + a) 结果相同(满足结合律)；但 - / / << >> 这类要选对左右。
      上面 print 的 (std::cout << ... << args) 正是【二元左折叠 (init op ... op pack)】
      (init=cout、op=<<、pack=args) → ((cout<<e1)<<e2)…<<en，左到右、按书写顺序输出。

    ── 空包(0 个参数)的规则：一元折叠多数非法 → 用带 init 的二元折叠 ────────────────
      一元折叠遇空包，【只有】&&(→true)、||(→false)、,(→void()) 合法；+/<< 等对空包【编译错误】。
      带 init 的二元折叠空包时结果= init，正好兜底：
        template<class... A> int sum(A... a) { return (0 + ... + a); }  // 二元左折叠，init=0
        sum();        // 空包 → = init = 0    （一元 (a + ...) 在空包时会编译错）
        sum(1, 2, 3); // → ((0+1)+2)+3 = 6
      ⟹ "可能传 0 个参数"时，用 ( init op ... op pack ) 给默认值。

    ── 逗号折叠 = 对每个元素做事(遍历)：(f(args), ...) ─────────────────────────────
      逗号运算符天然【从左到右求值】，所以 (f(args), ...) 与 (..., f(args)) 对执行顺序无影响；
      这是"遍历变参/对每个元素操作"的常用写法(见上一条"逐个循环处理")。

  - 对变参【逐个循环处理】：形参包【不是运行期容器】，不能直接对它写 for；常用【逗号折叠】把
    一段操作对每个元素【依次执行一遍】，效果就是"遍历变参"：
        template <typename... Args>
        void for_each_arg(const Args&... args) {
            // 逗号折叠 (expr, ...)：对每个 arg 依次求值一次 expr —— 等价于"循环遍历"
            ((std::cout << "[" << args << "] "), ...);
            std::cout << '\n';
        }
        for_each_arg(1, "hi", 3.14);     // → [1] [hi] [3.14]

    想要"下标"就再配个计数器（每次 i++）：
        template <typename... Args>
        void dump(const Args&... args) {
            int i = 0;
            ((std::cout << i++ << "=" << args << "  "), ...);   // 0=1  1=hi  2=3.14
        }

    若【所有参数同类型】，可用 {args...} 造出 initializer_list，再用 range-for【真正地循环】：
        template <typename... Args>
        int sum(Args... args) {                 // 假设都是 int
            int s = 0;
            for (int v : {args...}) s += v;     // {args...} 是 std::initializer_list<int>
            return s;
        }
        sum(10, 20, 30);                 // → 60

    (C++17 前无折叠，对每个元素做事常用 initializer_list + 逗号的"假循环"：
        int dummy[] = { 0, ((std::cout << args << ' '), 0)... }; (void)dummy; )

  - C++17 前没有折叠，用【递归】展开(一个 + 一包，剥洋葱)：
        void print() {}                                   // 递归终点
        template <class T, class... Rest>
        void print(const T& first, const Rest&... rest) {
            std::cout << first;
            print(rest...);                               // 把剩下的继续传
        }

  ★ 优点：【类型安全 + 各求值一次】
    - 每个实参有【推导出的类型】，编译期检查；类型不对直接编译报错，不像 C 变参那样运行期 UB。
    - 每个实参在调用处【求值一次】，按值或按引用绑定——没有宏那种"文本被多次展开、多次求值"的坑。
  - 本仓库实例：c10::str(args...) 就是变参模板——把任意多个、任意可 << 的实参拼成 std::string
    (见 torch_check_macro_analysis.txt 第五节(2))；TORCH_CHECK 的消息最终就交给它。


================================================================================
三·补、实战实例：c10::str 与 torchCheckMsgImpl 的实现 + 调用追踪
================================================================================

  把第三节的"变参模板 + 递归展开 / 重载决议"落到 PyTorch 的真实代码：c10::str（变参模板拼字符串）
  与 torchCheckMsgImpl（变参模板 + 重载分流）。它们正是 TORCH_CHECK 拼错误消息所用。

  ▶ c10::str —— 变参模板拼字符串（正是第三节"递归展开"的真实案例，简化）
        namespace detail {
          // 基例：把一个实参 << 进流（递归终点）
          template <typename T>
          std::ostream& _str(std::ostream& ss, const T& t) { ss << t; return ss; }
          // 递归：先处理 t，再处理剩下的 args（剥洋葱，每层只剥一个）
          template <typename T, typename... Args>
          std::ostream& _str(std::ostream& ss, const T& t, const Args&... args) {
              return _str(_str(ss, t), args...);
          }
        }
        template <typename... Args>
        std::string str(const Args&... args) {
            std::ostringstream ss;
            detail::_str(ss, args...);   // 把所有实参依次 << 进 ss
            return ss.str();             // ← 这里有【堆分配】，返回最终字符串
        }
      - 类型驱动：每个实参按【自己的静态类型】选 operator<<，无占位符、无需 %lld/强转。
      - 递归终止靠【重载偏序】：剩一个参数时，基例 _str(ss,t) 比递归版(空 Args)更具体而胜出。
      - 真实 c10 还对【0 参 / 单个 std::string / 单个 const char*】做特化，绕开 ostringstream/分配。

  ▶ torchCheckMsgImpl —— 变参模板 + 重载决议在编译期分流（简化）
        namespace c10::detail {
          // ① 变参模板：首参=默认串(忽略)，其余=用户消息 → 交给 c10::str
          template <typename... Args>
          auto torchCheckMsgImpl(const char* /*msg*/, const Args&... args) { return ::c10::str(args...); }
          // ② 只有默认串、无用户参数 → 返回默认串
          inline const char* torchCheckMsgImpl(const char* msg) { return msg; }
          // ③ 默认串 + 恰好一个 const char* → 直接返回该用户串（不拼接、不分配）
          inline const char* torchCheckMsgImpl(const char* /*msg*/, const char* args) { return args; }
        }
      非模板(②③)比模板(①)更优先：0 参→②默认串；单个 const char*→③原样；其余→①调 c10::str。

  ▶ 使用 + 调用追踪：TORCH_CHECK(ok, "half=", half)（设 int64_t half=16，ok 为假）
      宏展开后要先算第 4 个实参（消息），逐层调用：
        torchCheckMsgImpl("Expected…", "half=", half)     // 3 实参 → 命中①模板
        └─ c10::str("half=", half)                        // 默认串因 /*msg*/ 被丢弃
           ├─ ostringstream ss;
           ├─ _str(ss, "half=", half)            // ss 后 2 个 → 递归重载
           │  └─ _str( _str(ss,"half=") , half )
           │     ├─ _str(ss,"half=")  [基例] → ss << "half="  → ss="half="
           │     └─ _str(ss, half)    [基例] → ss << 16       → ss="half=16"
           └─ return ss.str()  →  "half=16"
      最终 torchCheckFail(__func__,__FILE__,__LINE__, "half=16") 抛 c10::Error → Python RuntimeError。
      要点：每层 _str 只剥一个参数(先 t 后 args)、按类型选 << 拼进同一个 ss；拼接顺序=书写顺序。

  ▶ 几种调用各走哪条（重载分流，编译期定）
        TORCH_CHECK(ok)                  → ② 默认串（不调 c10::str、不分配）
        TORCH_CHECK(ok, "bad")           → ③ 直接用 "bad"（不调 c10::str、不分配）
        TORCH_CHECK(ok, "half=", half)   → ① c10::str("half=", half) = "half=16"（分配）
        TORCH_CHECK(ok, n)  (n 是 int)   → ① c10::str(n)（单参但非 const char* → 仍走①）
      ⟹ "无消息 / 单字符串"是廉价路径(不分配)；只有"多段/带类型"才动用 c10::str 的 ostringstream。
      (更细的实现解析见 torch_check_macro_analysis.txt 第三节(2)、第五节(2)。)


================================================================================
四、与宏变参 __VA_ARGS__ 的对比（传文本 vs 传值）
================================================================================

  最本质的差别：宏传【源码文本】，函数(C 变参 / 变参模板)传【求值后的值】。

  ┌────────────────┬─────────────────────┬──────────────────────┬───────────────────────┐
  │                │ 宏变参 __VA_ARGS__   │ C 风格变参 ...(va_list)│ C++ 变参模板          │
  ├────────────────┼─────────────────────┼──────────────────────┼───────────────────────┤
  │ 传什么         │ 实参的【源码文本】   │ 求值后的值(从栈取)    │ 求值后的值(按值/引用) │
  │ 处理阶段       │ 预处理(编译前)       │ 运行期               │ 编译期实例化          │
  │ 有类型吗       │ 无(文本)             │ 无(va_arg 手动指定)   │ 有(类型推导)          │
  │ 类型安全       │ 无                  │ 无(易 UB)             │ 有                    │
  │ 求值次数       │ 0 次或多次(文本展开) │ 一次                  │ 一次                  │
  │ 能接"类型名"   │ 能(纯文本)           │ 不能                  │ 不能(只传值)*         │
  │ 能 #/## 文本操作│ 能(#字符串化/##拼接) │ 不能                  │ 不能                  │
  │ 取个数         │ 较繁(需技巧)         │ 靠约定(如 count/格式串)│ sizeof...(args) 直接   │
  │ 典型代表       │ TORCH_CHECK          │ printf               │ c10::str              │
  └────────────────┴─────────────────────┴──────────────────────┴───────────────────────┘
    * 变参模板传的是【值】；但"类型"可经【模板类型参数 typename... Ts】表达，那是另一种用法
      （传类型给模板），不是把"类型名当函数实参"。

  由"传文本 vs 传值"引出的关键差异：
    - 宏能玩文本：# 字符串化、## 拼接、接收类型名/代码片段（X-macro）——因为它操作的是源码文本。
    - 函数(尤其变参模板)换来类型安全 + 单次求值：实参先求值成带类型的值，编译期检查、不会多次求值。
    例：宏的多次求值坑 vs 模板的单次求值
        #define TWICE(x) ((x) + (x))
        TWICE(i++)          // → ((i++) + (i++))   i++ 执行两次！
        template<class T> T twice(T x){ return x + x; }
        twice(i++);         // i++ 先求值一次得到值，再 x+x —— 只自增一次


================================================================================
五、怎么选
================================================================================

  - 现代 C++、要类型安全、拼接/转发任意参数 → 用【C++ 变参模板】(如 c10::str、std::make_unique、
    转发构造)。首选。
  - 必须对接 C ABI / 老接口(printf 家族、回调) → 才用【C 风格变参】，并务必靠格式串/计数约定，
    注意 UB。
  - 需要在【文本/记号层】操作——字符串化(#)、拼接标识符(##)、接收类型名、条件编译、生成代码、
    或要捕获调用点 __FILE__/__LINE__ → 才用【宏】(如 TORCH_CHECK、X-macro、AT_DISPATCH_*)。


================================================================================
六、小结
================================================================================

- "变参"不是宏独有：C 风格变参(... + va_list)、C++ 变参模板(typename... Args)都能接任意多参数。
- 本质区别：宏传【源码文本】(预处理、无类型、可多次求值、能 #/## 玩文本)；函数传【求值后的值】
  (C 变参运行期取值、无类型安全；变参模板编译期、有类型、单次求值)。
- 选择：现代 C++ 优先【变参模板】(安全)；对接 C 用【C 变参】；要操作文本/类型名/字符串化/拼接
  才用【宏】。
- 实例：printf(C 变参)、c10::str(变参模板)、TORCH_CHECK(宏变参)。
