struct / class：类型、初始化、构造函数、成员的链接性与 inline
================================================================================

  src/inference/type_convert.cuh 中的 _typeConvert trait 模板：

      template <typename torch_type>
      struct _typeConvert { static constexpr bool exists = false; };

      template <> struct _typeConvert<float> {
        static constexpr bool exists = true;
        using hip_type = float;
        using packed_hip_type = float2;
        __device__ static __forceinline__ float convert(hip_type x){ return x; }
        ...
      };

================================================================================
一、struct 只含内置成员，编译器会从 0 赋值吗？—— 不会
================================================================================

  常见误解：把 struct 和 enum 记混。
    - enum 的枚举值才默认从 0 递增。
    - struct 的成员【不会】被编译器自动从 0 赋值。

      enum Color { Red, Green, Blue };   // Red=0, Green=1, Blue=2（这才是"从0开始"）

  struct 是否清零，取决于【如何创建/初始化对象】，与"成员是不是内置类型"无关：

    struct Point { int x; int y; };

    | 写法                      | 结果                                  |
    |---------------------------|---------------------------------------|
    | Point p;（函数内局部）    | ✗ 不初始化，x/y 是栈上垃圾值          |
    | Point p{}; / Point p = {};| ✓ 值初始化 → x=0, y=0                 |
    | Point p{1};               | x=1，其余补 0（y=0）                   |
    | Point p = {1, 2};         | x=1, y=2                              |
    | static Point p;（全局/静态）| ✓ 零初始化 → x=0, y=0               |
    | new Point（裸 new）       | ✗ 不初始化，垃圾值                    |
    | new Point()（带括号）     | ✓ 值初始化 → 0                        |

  结论：对内置类型成员，"自动清零"只发生在
    （a）静态/全局存储期对象，或（b）显式写了 {} / () 触发值初始化。
  函数里 Point p; 这种局部对象，成员是未定义的垃圾值，绝不是 0。


================================================================================
二、struct 没方法就不是类类型吗？没写构造函数就没构造函数吗？—— 都不对
================================================================================

  1. struct 永远是"类类型"
  --------------------------------------------------------------------------------

    C++ 里 struct 和 class 是同一种东西，唯一区别是默认访问权限
    （struct 默认 public，class 默认 private），其余完全等价。

        struct Point { int x; int y; };          // 类类型
        class  Point { public: int x; int y; };   // 完全等价

    → 只要是 struct，无论有无方法、成员是不是内置类型，它都是类类型。

  2. 编译器仍会隐式声明一整套特殊成员函数
  --------------------------------------------------------------------------------

    即使一个函数都没写，编译器也会为 struct Point 隐式声明：

        Point();                        // 默认构造
        Point(const Point&);            // 拷贝构造
        Point(Point&&);                 // 移动构造
        Point& operator=(const Point&); // 拷贝赋值
        Point& operator=(Point&&);      // 移动赋值
        ~Point();                       // 析构

    所以 "没写构造函数 = 没有构造函数" 是错的。默认构造等是有的，
    只是它 trivial（平凡）：对内置成员【不做任何初始化】
    （正是第一节"局部对象成员是垃圾值"的原因）。

  3. 它有个名字叫 aggregate（聚合体）
  --------------------------------------------------------------------------------

    "只有 public 数据成员、没有用户声明的构造函数、没有基类、没有虚函数"
    的类型称为 aggregate，可用花括号聚合初始化：

        Point p = {1, 2};
        Point q{1, 2};

    注意：aggregate 不是"没有构造函数"，而是"没有【用户自定义】构造函数"。


================================================================================
三、什么是构造函数？聚合体也有默认构造函数吗？
================================================================================

  构造函数是什么
  --------------------------------------------------------------------------------

    构造函数 = 创建对象时自动调用、负责把内存初始化成可用对象的特殊成员函数。
      - 名字与类名相同，没有返回类型（连 void 都没有）。
      - 不能手动调用，创建对象那一刻由编译器自动调用。
      - 可带参数，决定如何初始化。

        struct Point {
          int x, y;
          Point() { x=0; y=0; }            // 默认构造（无参）
          Point(int a, int b){ x=a; y=b; }  // 带参构造
        };

        Point p;        // 自动调 Point()     → x=0, y=0
        Point q(3, 4);  // 自动调 Point(3,4)  → x=3, y=4

    "默认构造函数"特指【不需要参数】就能调用的那个。

  聚合体也有默认构造函数吗？—— 有
  --------------------------------------------------------------------------------

    即使一行构造函数都没写，编译器也会隐式声明默认构造函数：

        struct Point { int x; int y; };
        Point p;     // ← 调用隐式默认构造（trivial，什么都不做 → x/y 垃圾值）

  关键区分：聚合初始化 ≠ 调用构造函数
  --------------------------------------------------------------------------------

        Point p;          // 调隐式默认构造（trivial，不清零）→ 垃圾值
        Point p{};        // 聚合初始化（值初始化）→ x=0, y=0
        Point p{3, 4};    // 聚合初始化，直接填 → x=3, y=4   ← 没调构造函数！

            Point p{3, 4};
                 │
                 ▼
        聚合初始化（aggregate init）：编译器直接 x←3, y←4，不调用任何构造函数

    "用户没写构造函数" ≠ "完全没有构造函数"——编译器隐式声明的默认构造仍存在。

四、静态成员函数：链接性、存储、访问、是否需类外声明
================================================================================

  以 __device__ static __forceinline__ float convert(...) 为例，澄清几个误解。

  1. "静态方法在静态存储区" —— 错，混了两个 static
  --------------------------------------------------------------------------------

    static 关键字严重重载，含义看用在哪：

    | 用法                        | 含义                                       |
    |-----------------------------|--------------------------------------------|
    | static 局部变量 / 成员变量  | 静态存储期——数据放 .data/.bss（静态存储区）|
    | static 成员函数（convert）  | 仅表示"无 this 指针、属于类而非实例"        |
    | static 自由函数（文件作用域）| 内部链接（internal linkage）               |

    关键：存储期是针对【对象/变量】的概念，函数根本没有存储期。
    函数机器码一律在【代码段 .text】。static 在成员函数上只意味"无 this、类级调用"。

  2. "静态方法是外链接" —— 对，但不是因为它 static
  --------------------------------------------------------------------------------

        static float f();        // 自由函数：static = 内部链接
        struct S {
          static float g();      // 成员函数：static ≠ 内部链接，仍是外部链接
        };

    convert 有外部链接是因为它是【成员函数】，与加不加 static 无关。
    又因它在类内定义 → 隐式 inline（见第五节）。

  3. 访问：类名::静态方法（无需对象）
  --------------------------------------------------------------------------------

        float r = vllm::_typeConvert<float>::convert(x);   // 类名::静态方法

  4. 类内定义了就不需类外声明/定义
  --------------------------------------------------------------------------------

        struct S { static float convert(float x){ return x; } };  // 类内定义→完事
        struct T { static float convert(float x); };              // 类内只声明
        float T::convert(float x){ return x; }                    // 才需类外定义

    _typeConvert 用的是前者（类内既声明又定义），无需类外。

  5. 什么需要类外定义？friend 吗？—— 典型是"非 inline 静态数据成员"
  --------------------------------------------------------------------------------

        struct S { static int count; };  // 类内：仅声明
        int S::count = 0;                // 类外：必须有这一处定义（否则链接 undefined）

      - C++17 起可 static inline int count = 0; 直接类内定义。
      - static constexpr 成员（如 _typeConvert::exists）自带 inline 语义，也无需类外。

    friend 不是"必须类外"：friend 声明本身不是成员，只授权访问；
    其函数可类内定义（隐藏友元）也可类外定义，两者皆可。


================================================================================
五、为什么类内定义的成员函数隐式 inline？类本身也隐式 inline 吗？
================================================================================

  先纠正前提：不是"static 函数"才隐式 inline，而是【任何在类定义体内给出定义的
  成员函数】都隐式 inline，与 static 无关。

  1. 原因：ODR + 头文件被多 TU 包含
  --------------------------------------------------------------------------------

    inline 的真正含义不是"展开优化"（那是早年提示，编译器现在自决），而是：
      允许该实体在多个翻译单元(TU)里出现【相同定义】，链接器视为同一个，不报重复定义。
    这是对 ODR（单一定义规则）的豁免。

    若不 inline 会怎样：

        // point.h（被 a.cpp、b.cpp 都 #include）
        struct Point { int x,y; int sum(){ return x+y; } };

    a.cpp、b.cpp 各得一份 Point::sum 定义：
      - 非 inline → 两个 TU 各产生一个 sum 符号 → 链接"重复定义"✗
      - 隐式 inline → ODR 豁免，多 TU 相同定义合法，链接器合并为一 ✓

        point.h 类内定义 sum()
             │ #include 进多个 .cpp
             ├── a.cpp → 一份 sum 定义
             └── b.cpp → 一份 sum 定义
                        非 inline → 重复定义 ✗ ；隐式 inline → 合并为一 ✓

    对比：类内只声明、类外定义的成员函数【不】隐式 inline，
    其定义只能放在某一个 TU（放头文件被多次包含就重复定义）。

  2. 类本身也是隐式 inline 吗？—— 不是，inline 不适用于类
  --------------------------------------------------------------------------------

    inline 只用于【函数】和（C++17 起）【变量】——会产生符号、可能重复定义的实体。
    类型不产生这种符号，没有 inline 概念。

    类能在多 TU 里定义（头文件被多次包含），靠的是 ODR 给【类型】的另一条专门豁免：
      同一个类允许在多个 TU 定义，只要每处定义【逐 token 完全相同】。

    | 实体              | 多 TU 多份定义靠什么合法                          |
    |-------------------|---------------------------------------------------|
    | 类 / 类型         | ODR 对类型的内建豁免（须逐字相同），不能也不需写 inline |
    | 函数 / 变量       | 需 inline（或类内定义自动 inline / C++17 inline 变量） |
    | 类内定义的成员函数| 自动 inline                                       |
    | 模板              | 模板有自己的 ODR 豁免（类外定义也能放头文件）     |


================================================================================
六、为什么静态数据成员需显式 inline 才能类内定义，否则必须类外定义？
================================================================================

  核心：函数是"代码"，静态数据成员是"对象(变量)"，两者 ODR 约束本质不同。

  1. 变量必须"全程序唯一、地址唯一"
  --------------------------------------------------------------------------------

    静态数据成员是个变量，占内存、有地址，整个程序里必须只有【一份实体】。
      - 函数是代码：inline 允许多 TU 各有相同定义，合并/挑一份，指向同一段代码。
      - 变量是对象：每个包含头文件的 TU 都"定义"一次 → 多个同名对象、多个地址，
        到底哪个是"那一个"？C++17 前【没有任何机制合并多份变量定义】。

    故传统做法（C++17 前）只能：

        struct S { static int count; };  // 类内：仅【声明】（承诺别处有定义）
        int S::count = 0;                // 类外：某一个 .cpp 给出唯一【定义】（分配存储）

  2. 函数没这问题：本就隐式 inline、可合并
  --------------------------------------------------------------------------------

    根子上：函数有 inline 合并机制，变量（C++17 前）没有。
    这就是函数能类内定义、静态数据成员当年不能的原因。

  3. C++17 的 inline 变量：把"合并机制"给了变量
  --------------------------------------------------------------------------------

        struct S { static inline int count = 0; };  // C++17：类内直接定义

    显式写 inline 是在主动声明"该变量采用多 TU 合并为一的链接语义"。
    为什么函数隐式、变量却要显式？类内定义函数语义无歧义，标准直接规定隐式 inline；
    变量则让你显式 opt-in，更稳妥。

  4. 特例：static constexpr 成员自动隐式 inline
  --------------------------------------------------------------------------------

        template <> struct _typeConvert<float> {
          static constexpr bool exists = true;   // C++17 起隐式 inline
        };

    static constexpr 数据成员自 C++17 起自动是 inline 变量：
    类内直接初始化即可，【既不用类外定义，也不用显式 inline】。
    这就是 _typeConvert::exists 被多个 .cu 包含也无链接冲突的原因。

七、非静态成员变量：不是 inline，为何也不需类外定义？
================================================================================

  非静态成员变量"inline 与否"这个问题根本不适用，且确实不需类外定义。

  核心：它没有"自己独立的存储"，只是对象的一部分
  --------------------------------------------------------------------------------

        struct Point {
          int x;   // 不是独立变量，只是"Point 对象里有个 int，在偏移 0"
          int y;   //              "再有个 int，在偏移 4"
        };

    类定义里写 int x; 没有创建任何对象、没分配任何内存，只是在【描述类型布局】。
    存储在创建对象时才分配：

        Point p;   // 此刻才分配 x、y（作为 p 的一部分）
        Point q;   // q 又有自己的一份 x、y

    成员存储期 = 所属对象的存储期（随对象生灭）。

  为什么不需类外定义
  --------------------------------------------------------------------------------

    因为根本没有"一个独立对象"需要定义：
      - 静态数据成员像"挂在类名下的全局变量"，有自己的静态存储 → 须一处唯一定义。
      - 非静态成员没有独立存储，内存由每个对象提供。类内 int x; 已把布局说清，
        创建对象时自然就有它的内存 → 无独立对象 → 无需类外定义。

  为什么 inline 不适用
  --------------------------------------------------------------------------------

    inline 解决"同一有链接实体在多 TU 多份定义如何合并"。非静态成员变量无链接、
    不产生独立符号，只是类型布局的一部分；类型在多 TU 重复定义靠 ODR 类型豁免
    （各 TU 逐字相同），int x; 随整个类一起重复即可，不参与 inline 机制。


================================================================================
八、非静态成员变量和局部变量一样，没有链接性吗？—— 是
================================================================================

  C++ 三种链接性
  --------------------------------------------------------------------------------

    链接性 = 一个名字能否在别的作用域/翻译单元里指向同一实体。

    | 链接性            | 含义                       | 典型例子                              |
    |-------------------|----------------------------|---------------------------------------|
    | 外部链接 external | 跨 TU 可见，产生全局符号    | 全局函数、非 static 全局变量、         |
    |                   |                            | 静态数据成员、成员函数                |
    | 内部链接 internal | 仅本 TU 可见                | 文件作用域 static 函数/变量、匿名命名空间 |
    | 无链接 no linkage | 仅在所在作用域可用，不产生  | 局部变量、函数参数、非静态数据成员、   |
    |                   | 可被别处引用的符号          | 局部类型                              |

    非静态数据成员与（非 static）局部变量都属【无链接】。

  为什么都"无链接"——本质相同
  --------------------------------------------------------------------------------

        int g;                 // 外部链接：别的 .cpp 用 extern int g; 可引用同一个 g
        void f(){ int local; } // 无链接：出了函数，名字 local 谁也引用不到
        struct Point { int x; };// 无链接：不能凭"x"从外部链接到它，只能 p.x 结构化访问

    成员 x 只能"穿过对象去取"（p.x、q.x 或成员函数内），不是靠名字的链接。
    无链接 → 不产生独立可链接符号 → 故不需类外/外部定义，也谈不上 inline。

  关键：链接性 ≠ 存储期（别混）
  --------------------------------------------------------------------------------

    | 实体                 | 链接性  | 存储期                                      |
    |----------------------|---------|---------------------------------------------|
    | 局部变量 int local;  | 无链接  | 自动存储期（进栈，离开作用域销毁）          |
    | 非静态成员 int x;    | 无链接  | 无独立存储期，随所属对象（栈/堆/静态都可能） |

    两者链接性相同，但局部变量有自己独立的栈存储，成员变量的存储寄生于宿主对象。

  对比：加 static 就跳到外部链接
  --------------------------------------------------------------------------------

        struct S {
          int x;          // 无链接（随对象）
          static int c;   // 外部链接 + 静态存储期 → 像"挂在类名下的全局变量"
        };
        int S::c = 0;     // 所以唯独它需要这一处定义来分配唯一的存储

九、总表
================================================================================

  | 实体                        | 链接性   | 存储/位置          | 需类外定义 | inline 概念 |
  |-----------------------------|----------|--------------------|------------|-------------|
  | 非静态数据成员 int x;       | 无链接   | 随对象（栈/堆/静态）| 否         | 不适用      |
  | 局部变量 int local;         | 无链接   | 自动（栈）          | 否         | 不适用      |
  | 静态数据成员 static int c;  | 外部链接 | 静态存储期(.data)  | 是(C++17前)| 适用(inline 变量) |
  | static constexpr 成员       | 外部链接 | 静态存储期          | 否         | 隐式 inline |
  | 类内定义的成员函数          | 外部链接 | 代码段 .text       | 否         | 隐式 inline |
  | 类内只声明、类外定义成员函数| 外部链接 | 代码段 .text       | 是         | 非 inline   |
  | static 自由函数（文件作用域）| 内部链接 | 代码段 .text       | —          | —           |
  | 类 / 类型                   | —        | 无运行期实体        | —          | 不适用(ODR 类型豁免) |

  贯穿全文的两条主线：
    1. 函数是"代码"（可经 inline 合并），数据成员/变量是"对象"
       （静态成员须全程序唯一，故 C++17 前必须类外定义一次）。
    2. "需不需类外定义 / 谈不谈 inline" 取决于实体【有没有自己独立的、
       可被链接引用的存储/符号】：无链接者（局部变量、非静态成员）天生不需要。
