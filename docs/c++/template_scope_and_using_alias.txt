模板形参与 using 类型别名的作用域隔离
================================================================================

一、问题背景
--------------------------------------------------------------------------------

  AT_DISPATCH_FLOATING_TYPES 展开的 case 块内有：
    using scalar_t = float;

  kernel 函数定义中有：
    template <typename scalar_t, typename scalar_t4, typename scalar_i>
    __global__ void softmax_kernel(scalar_t* a, scalar_t* b, ...);

  两处均使用名字 scalar_t，是否冲突？


二、结论
--------------------------------------------------------------------------------

  不冲突。两个 scalar_t 在完全独立的作用域内，编译器分别解析，互不影响。


三、两个 scalar_t 的作用域
--------------------------------------------------------------------------------

  【using scalar_t = float】

    作用域：case 块的 { } 内（局部类型别名）
    性质：编译期类型别名，在当前 { } 结束时消亡
    来源：AT_DISPATCH_FLOATING_TYPES 宏注入

    case at::ScalarType::Float: {
        using scalar_t = float;    ← 作用域开始
        softmax_kernel<scalar_t><<<...>>>();
                       ↑ 此处的 scalar_t 查找到 float
    }                              ← 作用域结束，scalar_t 不再可见

  【template <typename scalar_t>】

    作用域：kernel 函数模板的定义体内
    性质：模板形参，实例化时由调用方传入的模板实参绑定
    来源：kernel 函数的模板声明

    template <typename scalar_t, typename scalar_t4, typename scalar_i>
    __global__ void softmax_kernel(scalar_t* a, scalar_t* b, ...) {
        // 这里的 scalar_t 是模板形参，由实例化时决定具体类型
    }


四、两者唯一的联系：模板实参传递
--------------------------------------------------------------------------------

  softmax_kernel<scalar_t><<<grid, block>>>(...);
  //              ↑
  //  编译器在 case 块作用域内查找 scalar_t → 找到 using 别名 → float
  //  将 float 作为模板实参传给 softmax_kernel 的第一个模板形参
  //
  //  等价于显式写：
  //  softmax_kernel<float><<<grid, block>>>(...);

  这一句完成了两个作用域之间唯一的交互：
    调用点的 scalar_t（using 别名 = float）
      → 作为模板实参传入
      → kernel 模板形参 scalar_t 实例化为 float

  之后两个作用域彻底分离，各自独立。


五、名字相同不造成冲突的原因
--------------------------------------------------------------------------------

  C++ 名字查找规则：在某处使用一个名字，编译器从当前位置向外逐层搜索，
  找到第一个匹配即停止，不会继续向外。

  在 case 块内使用 scalar_t：
    查找路径：case 块 → 找到 using scalar_t = float → 停止
    不会查找到 kernel 函数的模板形参（两者不在同一查找链上）

  在 kernel 函数体内使用 scalar_t：
    查找路径：函数体 → 函数模板形参 → 找到 typename scalar_t → 停止
    不会查找到 case 块的 using 别名（case 块已不在作用域内）

  两个同名 scalar_t 处于完全不相交的查找链，编译器永远不会将二者混淆。


六、名字相同是约定俗成
--------------------------------------------------------------------------------

  kernel 模板形参与 AT_DISPATCH 注入的别名都叫 scalar_t，
  是 PyTorch 生态的命名惯例，目的是让代码可读：
    "这个类型是张量的标量元素类型"

  改成不同名字（如 T）完全合法，效果完全相同：

    // 调用点
    using scalar_t = float;
    softmax_kernel<scalar_t><<<...>>>();   // 传入 float

    // kernel 定义（改名为 T）
    template <typename T, typename T4, typename Ti>
    __global__ void softmax_kernel(T* a, T* b, ...) { ... }
    // 实例化为 softmax_kernel<float>，T = float，行为完全一致

  名字不同不影响任何语义，改名只影响可读性。


七、汇总
--------------------------------------------------------------------------------

  | 元素                    | 作用域         | 性质       | 值/类型               |
  |-------------------------|---------------|------------|-----------------------|
  | using scalar_t = float  | case 块 { }   | 编译期别名 | float（固定）         |
  | template <typename scalar_t> | kernel 函数体 | 模板形参 | 实例化时由实参决定  |
  | softmax_kernel<scalar_t> | case 块内调用 | 模板实参   | 取 using 别名的值传入 |

  两个 scalar_t 名字相同，作用域完全不相交，编译器独立解析，互不干扰。
  唯一的联系是调用处的 <scalar_t> 把 float 作为模板实参传入 kernel 完成实例化。

附一、为什么先写"通用主模板"，再写每个特化——能否直接写特化？
================================================================================

  代码（src/inference/type_convert.cuh）：

    template <typename torch_type>
    struct _typeConvert {                 // 主模板：默认"不支持"
      static constexpr bool exists = false;
    };

    template <>
    struct _typeConvert<float> {          // 特化：float
      static constexpr bool exists = true;
      using hip_type = float;
      using packed_hip_type = float2;
      __device__ static __forceinline__ float convert(hip_type x) { return x; }
      ...
    };
    template <> struct _typeConvert<c10::Half>     { ... };  // 特化：Half
    template <> struct _typeConvert<c10::BFloat16> { ... };  // 特化：BFloat16

  问题：为什么不能省掉主模板，直接写各个特化？


  一、语法刚性要求：特化必须依附于已声明的主模板
  --------------------------------------------------------------------------------

    template <> struct _typeConvert<float> {...} 叫"显式（全）特化"。
    "特化"的含义是：对一个【已存在的模板】，在特定模板实参下给出特殊版本。
    编译器看到 _typeConvert<float> 时，必须先知道
    "_typeConvert 是一个接受一个类型参数的模板"，才能理解 <float> 是在特化它。

    若删掉主模板，只写：

        template <> struct _typeConvert<float> { ... };   // ✗ 编译错误

    报错：'_typeConvert' is not a template /
          explicit specialization of non-template。

    → 主模板是"被特化对象"的声明，是特化成立的语法前提。
      "只写特化、不写主模板"本身就不合法。


  二、语义作用：主模板提供"默认 / 兜底"行为（白名单机制）
  --------------------------------------------------------------------------------

    主模板故意只声明 exists=false，不定义 hip_type / convert：

        template <typename torch_type>
        struct _typeConvert { static constexpr bool exists = false; };

    配合 kernel 中：

        using Converter = vllm::_typeConvert<scalar_t_in>;
        static_assert(Converter::exists, "... data type is not supported ...");
        using T_in = typename Converter::hip_type;

    匹配逻辑：

        scalar_t_in = float / Half / BFloat16
              │  命中对应特化
              ▼
        exists = true → static_assert 通过，可取 hip_type / convert

        scalar_t_in = 其它类型（int8、double，或 sm_80 以下的 BF16）
              │  无特化，落到主模板
              ▼
        exists = false → static_assert 在【编译期】给出清晰报错

    这正是 type_convert.cuh 注释所述：
      "Each struct should have member static constexpr bool exists:
       If false, the optimized kernel is not used for the corresponding type."


  三、对比：三种写法在"传入未支持类型"时的差异
  --------------------------------------------------------------------------------

    | 写法                                 | 未支持类型传入时    | 报错信息          |
    |--------------------------------------|---------------------|-------------------|
    | 主模板带 exists=false（本项目）      | static_assert 失败  | 自定义、可读      |
    | 主模板只声明不定义                   | 用到不完整类型      | incomplete type   |
    |   template<typename T> struct X;     |                     | （晦涩）          |
    | 完全不写主模板                       | ——                  | 根本编译不过      |
    |                                      |                     | （非模板）        |

    本项目选第一种，是为了用 static_assert 给出【人类可读】的诊断信息，
    并允许 if constexpr (Converter::exists) 之类的编译期分支，而非硬报错。


  四、小结
  --------------------------------------------------------------------------------

    1. 语法：特化必须依附已声明的主模板，"直接写特化"编译不过。
    2. 设计：主模板的 exists=false 是白名单的默认分支——没专门特化的类型
       一律视为"不支持"，再由 static_assert 编译期友好报错。

    这是 trait 模板的标准写法：
      主模板定义"通用契约 + 默认答案"，特化只覆盖真正支持的少数类型。

附二、using namespace vllm::cuda_async; 是什么意思
================================================================================

  代码（src/inference/fused_qknorm_rope_kernel.cu:91）：

    using namespace vllm::cuda_async;

  含义：这是一条 using 指令（using-directive），把嵌套命名空间
        vllm::cuda_async 里的所有名字"摊开"到当前作用域，
        之后【不用写前缀】即可直接调用其中的函数。


  一、它在做什么
  --------------------------------------------------------------------------------

    async_util.cuh 把一组 CUDA 异步拷贝设备端工具函数包在两层命名空间里：

        namespace vllm {
        namespace cuda_async {
          cp_async_shared_global_16_cg(...)   // cp.async 16 字节拷贝
          cp_async_shared_global_ca(...)      // cp.async 任意大小拷贝
          cp_async_commit_group()             // cp.async.commit_group
          cp_async_wait_group<n>()            // cp.async.wait_group
        }  // namespace cuda_async
        }  // namespace vllm

    它们的全名很长（vllm::cuda_async::cp_async_commit_group()）。
    写了 using namespace vllm::cuda_async; 后，kernel 里可直接：

        cp_async_commit_group();
        cp_async_wait_group<0>();
        cp_async_shared_global_16_cg(smem, gmem);


  二、三种写法对比
  --------------------------------------------------------------------------------

    | 写法                                          | 调用时                          |
    |-----------------------------------------------|---------------------------------|
    | 什么都不加                                    | vllm::cuda_async::cp_async_...() |
    | using namespace vllm::cuda_async;（本文件）   | cp_async_...()  ← 全摊开         |
    | using vllm::cuda_async::cp_async_commit_group;| 只摊开这一个名字（using 声明）  |


  三、关键区别：不是命名空间别名 namespace X = Y
  --------------------------------------------------------------------------------

      using namespace vllm::cuda_async;   // using 指令：把里面的名字引入当前作用域
      namespace ca = vllm::cuda_async;    // 别名：给命名空间起短名，仍要写 ca::xxx

    前者让你【省掉前缀】；后者只是【缩短前缀】。


  五、命名空间别名只能用 namespace，不能用 using
  --------------------------------------------------------------------------------

    给命名空间起短名，只能写 namespace 别名，不能用 using 别名声明：

        namespace ca = vllm::cuda_async;   // ✓ 命名空间别名（namespace alias）
        using ca = vllm::cuda_async;       // ✗ 编译错误

    原因：两条语法的对象不同。
      - namespace 别名：右边必须是一个【命名空间】，专给命名空间起短名。
      - using 别名声明（alias declaration）：右边必须是一个【类型】，
        如 using u32 = uint32_t; / using Vec = std::vector<int>;。

    vllm::cuda_async 是命名空间、不是类型，故 using ca = vllm::cuda_async; 报错：

        error: expected a type, got 'vllm::cuda_async'
        error: 'vllm::cuda_async' is a namespace, not a type

    对照记忆：

      | 想别名的东西 | 正确语法              | 例子                               |
      |--------------|-----------------------|------------------------------------|
      | 命名空间     | namespace 别名 = ...; | namespace ca = vllm::cuda_async;   |
      | 类型         | using 别名 = ...;     | using T_in = __half;               |
      | 类型（旧式） | typedef ... 别名;     | typedef __half T_in;               |

    三者勿混（同名 using 但语义完全不同）：

        using namespace vllm::cuda_async;   // using 指令：摊开里面所有名字，调用免前缀
        namespace ca = vllm::cuda_async;    // 命名空间别名：缩短前缀，写 ca::cp_async_...()
        using T = some_type;                // 别名声明：给"类型"起别名（与命名空间无关）


  四、为什么这里这样用没问题
  --------------------------------------------------------------------------------

    using namespace 一般不建议放在头文件全局作用域（会污染所有包含它的文件），
    但此处：

      1. 写在 .cu 实现文件里，不是头文件 → 污染范围仅限本翻译单元。
      2. cuda_async 里的名字带专属前缀（cp_async_*），不易与其它符号冲突。

    所以这是图方便、可接受的局部用法——让 kernel 里 cp.async 流水线调用
    写起来更短、更易读。

附三、_typeConvert 是结构体，为什么也能用 using 起别名？
================================================================================

  代码（src/inference/fused_qknorm_rope_kernel.cu）：

    using Converter = vllm::_typeConvert<scalar_t_in>;

  疑问：_typeConvert 是 struct（结构体），为什么也能 using？


  一、关键：using 别名声明的右边要求是"类型"，而 struct 就是一种类型
  --------------------------------------------------------------------------------

    using 别名声明（alias declaration）右边必须是一个【类型】。
    C++ 里"类型"不只有 int / float 这种内置类型，还包括类类型：

        using A = int;                       // 内置类型
        using B = std::vector<int>;          // 类模板实例化后的类型
        using C = vllm::_typeConvert<float>; // struct 模板实例化后的类型 ← 同理

    struct / class 定义出来的就是【类类型（class type）】。
    _typeConvert<scalar_t_in> 是模板 _typeConvert 用 scalar_t_in 实例化后得到的
    一个具体类类型，所以它是合法的 using 别名目标。


  二、不要和"命名空间别名"那条混了（参见附二第五点）
  --------------------------------------------------------------------------------

        using Converter = vllm::_typeConvert<scalar_t_in>;   // ✓ 右边是 struct 类型
        using ca        = vllm::cuda_async;                  // ✗ 右边是命名空间

    上一处之所以不行，是因为 cuda_async 是命名空间、不是类型。
    唯一判断标准就是【右边是不是一个类型】：

      | 右边是什么                     | 能否用 using X = ...            |
      |--------------------------------|---------------------------------|
      | struct / class（含模板实例化） | ✓ 可以                          |
      | 内置类型、指针、引用、函数类型 | ✓ 可以                          |
      | 命名空间                       | ✗ 不行，要用 namespace X = ...  |
      | 函数、变量、对象               | ✗ 不行（那是值/实体，不是类型） |


  三、这里 using Converter = ... 起的作用
  --------------------------------------------------------------------------------

    _typeConvert<scalar_t_in> 这个类型名又长又带模板参数，起别名后代码更短，
    而且能从它里面取嵌套成员：

        using Converter = vllm::_typeConvert<scalar_t_in>;
        static_assert(Converter::exists, "...");        // 取静态成员
        using T_in  = typename Converter::hip_type;     // 取嵌套类型
        using T2_in = typename Converter::packed_hip_type;
        Converter::convert(x);                          // 调静态方法

    若不起别名，每处都得写全 vllm::_typeConvert<scalar_t_in>::hip_type，啰嗦且易错。
    所以 using Converter = ... 纯粹是给"这个 struct 类型"取个短名，
    和给 int 取别名没有本质区别。

附四、为什么 typename Converter::hip_type 要加 typename？
================================================================================

  代码（src/inference/fused_qknorm_rope_kernel.cu）：

    template <typename scalar_t_in, ...>
    __global__ void fusedQKNormRopeKernel(...) {
      using Converter = vllm::_typeConvert<scalar_t_in>;   // 依赖模板参数
      static_assert(Converter::exists, "...");             // 取"值"，不加 typename
      using T_in  = typename Converter::hip_type;          // 取"类型"，必须加 typename
      using T2_in = typename Converter::packed_hip_type;   // 同上
      ...
    }

  疑问：T_in / T2_in 这两行为什么要在前面加 typename？


  一、根源：依赖名（dependent name）+ 模板的两阶段解析
  --------------------------------------------------------------------------------

    Converter = vllm::_typeConvert<scalar_t_in>，它【依赖模板参数 scalar_t_in】，
    因此是一个"依赖类型"。

    编译模板【定义】时，scalar_t_in 还没确定具体类型，编译器看不到 Converter 内部
    到底有什么——_typeConvert<float> 与其它特化的 hip_type 可能完全不同，
    甚至某个特化里 hip_type 也可能是个静态成员（值）而非类型。

    像 Converter::hip_type 这种"通过依赖类型访问的成员名"叫【依赖名】。
    C++ 规定：

        编译器遇到 T::x 这种依赖名时，默认假定 x 是一个【值】
        （成员变量 / 静态成员 / 函数），而不是类型。

    所以要把它当类型用，必须显式写 typename 来纠正这个默认假定。


  二、不加 typename 会怎样
  --------------------------------------------------------------------------------

        using T_in = Converter::hip_type;   // ✗ 编译器把 hip_type 当"值"
                                            //   而 using T_in = 一个值; 不合法 → 报错

    典型报错：

        error: need 'typename' before 'Converter::hip_type' because
               'Converter' is a dependent scope

    加上 typename 后，编译器确知它是类型，using T_in = 类型; 成立：

        using T_in = typename Converter::hip_type;   // ✓


  三、对比：什么时候【不】用加
  --------------------------------------------------------------------------------

    作用域【不依赖】模板参数时，编译器当场就能查出它是不是类型，无需 typename：

        using A = std::vector<int>::value_type;          // ✓ 不用（具体类型）
        using B = typename std::vector<U>::value_type;   // ✓ U 是模板参数 → 依赖 → 要加

    | 场景                                  | 是否加 typename |
    |---------------------------------------|-----------------|
    | 具体类型::成员类型（不依赖模板参数）  | 不用            |
    | 依赖类型::成员类型（依赖模板参数）    | 必须加          |
    | 取的是值成员（如 Converter::exists）  | 不加（本就是值）|


  四、与本项目代码的对应
  --------------------------------------------------------------------------------

        static_assert(Converter::exists, "...");        // exists 是值(静态成员) → 不加
        using T_in  = typename Converter::hip_type;     // hip_type 是类型 → 必须加
        using T2_in = typename Converter::packed_hip_type;

    同一个依赖类型 Converter，取【值成员】exists 不加 typename，
    取【类型成员】hip_type / packed_hip_type 必须加。这正好印证规则：
    typename 专门用于在依赖作用域里，把"默认当作值"的成员名【纠正为类型】。


  五、一句话
  --------------------------------------------------------------------------------

    Converter 依赖模板参数、是依赖类型；Converter::hip_type 这种依赖名编译器
    默认当成"值"，必须用 typename 显式声明它是"类型"，using 才能成立；
    而 exists 本就是值，故不加。

    （延伸：模板里取"依赖模板"的成员模板时，对应还需 template 消歧关键字，
      如 obj.template foo<int>()；与此处 typename 是同一类"消歧"机制。）
